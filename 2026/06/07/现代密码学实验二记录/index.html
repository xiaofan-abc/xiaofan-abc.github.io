Question 2：Byte-at-a-time ECB decryption (Harder)

题目来源：

题目要求在加密 oracle 前面加入未知长度的随机前缀，然后仍然恢复 oracle 追加的未知字符串。相比普通 byte-at-a-time ECB 攻击，难点是攻击者输入不再天然从块边界开始。

解题思路

程序中的 oracle 形式为：

核心步骤如下：

  1. 通过不断增加输入长度，检测密文长度第一次增长的幅度，得到 block size 为 16。
  2. 输入大量相同字节，检查是否出现重复密文块，确认 oracle 使用 ECB。
  3. 构造两块连续相同的 marker，枚举前置填充长度，找到能让 marker 对齐到块边界的位置。
  4. 跳过随机前缀占用的密文块，对未知字符串做字节级字典攻击。

字典攻击的关键是：当需要恢复第 n 个字节时，先让目标字节落在某个块的最后一位，再枚举 0-255，比较目标块密文是否相等。

遇到的问题与解决

第一，MTC3 题里缺失的 MRZ 字符不能随便枚举，需要先理解 MRZ 校验位规则。解决方法是按 7, 3, 1 权重恢复有效期字段校验位。

第二，Cryptopals 14 的随机前缀会破坏输入对齐。解决方法是通过两块相同 marker 找到对齐所需的填充长度，并记录需要跳过的前缀密文块数。

第三，CBC bit flipping 里注入目标字符串会破坏上一块明文，这是 CBC 可塑性的正常副作用。实际判断只要求解密结果中包含 ;admin=true;，不要求所有块都保持可读。

实验总结

实验二集中体现了分组密码工作模式的安全边界。ECB 的确定性会泄露相同明文块的位置，使得 oracle 场景下可以逐字节恢复未知明文；CBC 本身虽然隐藏了相同明文块，但如果没有完整性校验，密文仍然具有可塑性，可以被攻击者翻转成有意义的控制字段。

通过这次实验，我更直观地理解了“加密不等于认证”。在真实系统中，单独使用 CBC 或 ECB 都不足以保证安全，必须结合随机 IV、正确 padding 校验和消息认证机制，或者直接使用 AEAD 模式。
