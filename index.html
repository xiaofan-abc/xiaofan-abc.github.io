实验总结

这个实验很适合作为 RSA 安全性的反例合集。1024 比特模数本身并不意味着万无一失，只要参数生成或使用方式存在问题，例如共模、共享素因子、低指数重复广播，攻击者就能避开通用大整数分解，直接恢复消息。

通过这次实验，我也更直观地理解了“参数管理”在公钥密码中的重要性。很多攻击并不是因为 RSA 数学公式错了，而是因为实现者在密钥生成、指数选择和消息发送策略上留下了结构性漏洞。

Question 2：Implement RSA

题目来源：

题目要求实现 RSA：生成公私钥，完成整数消息的加密和解密。实验中使用固定素数保证结果可复现。

解题思路

RSA 的核心步骤如下：

  1. 选择两个素数 p 和 q，计算 n = pq。
  2. 计算欧拉函数 φ(n) = (p - 1)(q - 1)。
  3. 选择公钥指数 e，要求 gcd(e, φ(n)) = 1。
  4. 用扩展欧几里得算法计算私钥指数 d = e^-1 mod φ(n)。
  5. 加密：c = m^e mod n。
  6. 解密：m = c^d mod n。

本实验使用 Python 的三参数 pow(base, exp, mod) 执行模幂运算，避免先计算巨大整数再取模。

遇到的问题与解决

第一，e 必须和 φ(n) 互素。最初随手选择素数时，e = 3 与 φ(n) 不互素，导致私钥指数不存在。解决方法是换用满足条件的素数对，并在代码中显式检查 gcd(e, φ(n))。

第二，RSA 直接加密的是整数代表，要求 0 <= m < n。如果明文转换成整数后大于等于 n，就不能直接加密。实验中使用短消息 hi 做演示，保证明文整数小于模数。

第三，Project Euler 182 如果直接对所有消息逐个验证会非常慢。使用未隐藏消息数量公式后，只需要枚举合法 e，复杂度大幅降低。

实验总结

实验三让我把 RSA 的两个侧面联系起来：一方面，RSA 加解密的实现依赖模逆、欧拉函数和快速模幂；另一方面，公钥指数的选择也会影响某些消息是否被真正隐藏。Project Euler 182 不是传统意义上的破译题，但它说明 RSA 参数选择并不只是“能解密”就够，还要关注数学结构带来的特殊消息。

Cryptopals 39 的实现则提醒我，课本上的 RSA 公式只是核心骨架。真实使用 RSA 时还必须考虑消息编码、padding、随机化和完整性保护，否则即使数学过程正确，也可能留下攻击空间。

遇到的问题与解决

第三，CBC bit flipping 里注入目标字符串会破坏上一块明文，这是 CBC 可塑性的正常副作用。实际判断只要求解密结果中包含 ;admin=true;，不要求所有块都保持可读。

实验总结

实验二集中体现了分组密码工作模式的安全边界。ECB 的确定性会泄露相同明文块的位置，使得 oracle 场景下可以逐字节恢复未知明文；CBC 本身虽然隐藏了相同明文块，但如果没有完整性校验，密文仍然具有可塑性，可以被攻击者翻转成有意义的控制字段。

通过这次实验，我更直观地理解了“加密不等于认证”。在真实系统中，单独使用 CBC 或 ECB 都不足以保证安全，必须结合随机 IV、正确 padding 校验和消息认证机制，或者直接使用 AEAD 模式。

遇到的问题与解决

1. repeating-key XOR 的 key size 不能只看一组块

如果只比较前两个块，偶然性较大，容易选到错误的 key size。解决方法是对多个块两两计算 Hamming distance，再求平均归一化距离。

2. single-byte XOR 需要评分函数

暴力枚举 0 到 255 的密钥字节后，需要判断哪个结果更像英文。实验中使用英文字母频率、空格、常见标点和不可打印字符惩罚构造评分函数。

3. SHA1 题不能直接暴力所有字符串

如果对所有字符集做无约束暴力搜索，搜索空间会非常大。题目实际给出了键盘指纹线索，因此应把搜索空间限制在 8 个物理按键及其 Shift 变体上。

实验总结

第一题体现了多表异或加密的典型攻击方式：先估计密钥长度，再把问题拆成多个 single-byte XOR。Hamming distance 和英文频率分析是破解 repeating-key XOR 的关键。

第二题体现了口令破解中“利用上下文线索缩小搜索空间”的重要性。目标哈希本身不可逆，但题目给出键盘指纹后，可以把原本很大的暴力空间缩小到约一千万个候选，并在较短时间内找到口令。

两道题都说明，密码算法或哈希函数本身可能没有被直接攻破，但弱密钥、可预测口令和额外线索会显著降低攻击难度。

终端中文字符乱码处理

在处理这个问题之前，你首先得知道为什么会出现这个这个问题。

你在使用VScode编辑代码时，代码页面中文正常，而终端输出那里中文却为乱码。

出现这个现象的原因是因为编码方式的不同。（VScode的默认编码方式为UTF-8，中国地区下cmd的编码方式GBK）

因为VScode终端那里调用的是cmd，两者编码方式的不同的就导致了中文乱码的问题。

所以我们解决乱码的方式，就是将两者的编码方式统一就行，要么将两者都统一为UTF-8，要么就统一为GBK。（个人建议统一为GBK）
